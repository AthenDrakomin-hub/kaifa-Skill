# 技术栈选型

## 任务目标
- 本Skill用于：基于项目需求推荐最合适的技术栈组合
- 能力包含：技术栈推荐、方案对比、约束分析、技术决策支持
- 触发条件：已完成产品概述，需要确定技术方案

## 前置准备
- 必须存在 `specs/产品概述.md`
- 工作目录：项目根目录

## 操作步骤

### 1. 读取产品概述
智能体扮演CTO角色，首先读取产品概述文档：
```
读取 specs/产品概述.md
```

### 2. 分析项目约束
基于产品概述分析以下约束：
- **项目规模**：小型/中型/大型
- **用户量级**：个人使用/小团队/公开服务
- **实时性要求**：静态内容/动态交互/实时通信
- **部署环境**：云平台/VPS/本地
- **开发周期**：快速原型/正式产品/长期项目
- **团队技能**：前端为主/全栈/后端为主

### 3. 生成技术栈推荐
基于约束分析，推荐完整的技术组合：

```markdown
# 技术栈选型

## 1. 技术约束分析
### 项目规模
[分析结论]

### 用户量级
[分析结论]

### 实时性要求
[分析结论]

### 部署环境
[分析结论]

## 2. 推荐方案

### 方案概述
[一句话描述推荐方案]

### 前端技术
- **框架**：[框架名称] - [选择理由]
- **UI库**：[库名称] - [选择理由]
- **状态管理**：[方案] - [选择理由]

### 后端技术
- **框架**：[框架名称] - [选择理由]
- **数据库**：[数据库名称] - [选择理由]
- **缓存**：[方案] - [选择理由]

### 部署方案
- **平台**：[平台名称] - [选择理由]
- **CI/CD**：[方案] - [选择理由]

### 开发工具
- **代码规范**：[工具] - [选择理由]
- **测试框架**：[框架] - [选择理由]

## 3. 方案对比

### 备选方案A
[描述方案A及其优劣]

### 备选方案B
[描述方案B及其优劣]

### 为什么选择推荐方案
[详细说明推荐理由]

## 4. 关键决策点
[列出需要用户确认的技术决策]

## 5. 风险评估
[列出可能的技术风险和应对方案]
```

### 4. 引导用户决策
对于关键决策点，提供选项让用户选择：

```
关于评论功能，有两种方案：

选项 A：使用第三方评论服务（Giscus）
优点：零成本、零维护、基于 GitHub Discussions
缺点：依赖第三方服务、用户需登录 GitHub

选项 B：自建评论系统
优点：完全自主控制、用户体验更好
缺点：开发工作量大、需要额外维护

我的建议：如果第一版要快速上线，选 A 更合适。你怎么看？
```

### 5. 保存文档
将技术栈选型结果保存到 `specs/技术栈.md`

## 输出格式
- 文件路径：`specs/技术栈.md`
- 文档格式：结构化的技术选型文档
- 内容要求：具体、有理有据、包含对比分析

## 注意事项

### 边界守卫规则
- **当前阶段只能做**：技术栈选型和技术方案对比
- **禁止行为**：不涉及具体代码编写、不涉及数据库表设计
- 如果用户问"代码怎么写"，提醒："我们先把技术选好，具体的代码实现后面会处理"

### 项目上下文协议
- 执行前必须读取 `specs/产品概述.md`
- 基于产品概述的需求进行技术选型

### 推荐原则
- 推荐"最合适"而不是"最热门"
- 考虑用户的技术背景和学习成本
- 优先选择生态成熟、文档完善的方案
- 考虑AI训练数据的丰富程度（AI是否熟悉该技术）

## 使用示例

**用户输入**：
```
/project-tech-stack 请进行技术选型
```

**智能体响应**：
```
我已经读取了你的《产品概述》，这是一个个人博客项目，核心需求是 Markdown 写作 + SEO 友好 + 低运维成本。基于分析，我推荐以下方案：

推荐方案：Next.js + Supabase + Vercel

前端/后端：Next.js 14（App Router）—— SSR 天然对 SEO 友好，前后端一把梭，不需要单独维护后端
数据库：Supabase（PostgreSQL）—— 免费额度足够个人博客用，自带用户认证
部署：Vercel —— Next.js 的亲儿子，推上去就能跑，零运维
样式：Tailwind CSS —— 开发快，AI 写起来也最顺手

有一个细节需要确认：你需要评论功能吗？

A：不需要，保持极简
B：需要，用第三方服务（如 Giscus，基于 GitHub Discussions，零成本）
C：需要，自建评论系统（工作量会增加不少）

我的建议：如果要做的话选 B，成本为零且不影响核心开发节奏。
```
