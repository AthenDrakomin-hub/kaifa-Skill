# 功能需求澄清

## 任务目标
- 本Skill用于：深入挖掘单个功能的需求细节和验收标准
- 能力包含：功能边界定义、用户场景分析、验收标准制定、需求文档生成
- 触发条件：用户说"我想做个XX功能"或路线图中的功能需要开始开发

## 前置准备
- 建议存在 `specs/产品概述.md`（了解项目上下文）
- 工作目录下将创建 `specs/features/` 目录存储功能文档

## 操作步骤

### 1. 读取项目上下文
智能体扮演产品经理角色，首先读取项目文档：
```
读取 specs/产品概述.md（如果存在）
读取 specs/开发路线图.md（如果存在）
```

### 2. 引导式需求挖掘
通过具体提问挖掘功能细节：

**第一轮：使用场景**
```
评论主要出现在什么地方？

A：文章详情页（读者对文章发表看法）
B：商品页面（买家评价商品）
C：其他场景

推荐：A，因为根据产品概述，你做的是内容平台
```

**第二轮：用户角色**
```
谁能评论？

A：所有访客（无需登录）
B：注册用户（需要登录）
C：特定权限用户

推荐：B，既能防止垃圾评论，又能建立用户体系
```

**第三轮：功能边界**
```
评论能做什么？

A：只能发表评论
B：可以回复评论
C：可以点赞评论
D：以上全部

推荐：D，完整功能，但我们可以分阶段实现
```

**第四轮：限制条件**
```
有什么限制？

A：评论字数限制
B：评论需要审核
C：禁止某些词汇
D：以上全部

推荐：A+C，先做基础限制，审核功能可以后期加
```

**第五轮：异常处理**
```
评论被删除后怎么处理？

A：显示"已删除"
B：完全移除，不显示
C：保留占位，但内容隐藏

推荐：A，告诉用户有评论被删除，避免困惑
```

### 3. 生成功能需求文档
基于对话生成标准化的需求文档：

```markdown
# [功能名称]功能需求

## 1. 功能概述
[一句话描述功能的核心价值]

## 2. 用户角色
- **主要用户**：[角色描述]
- **权限区分**：[不同角色的权限]

## 3. 使用场景
### 场景一：[场景名称]
- **触发条件**：[什么情况下触发]
- **操作流程**：
  1. 用户进入[页面]
  2. 用户执行[操作]
  3. 系统响应[结果]
- **预期结果**：[用户期望的结果]

### 场景二：[场景名称]
[同上]

## 4. 功能需求

### FR-01：[需求名称]
- **描述**：[详细描述]
- **优先级**：高/中/低
- **关联场景**：场景X

### FR-02：[需求名称]
[同上]

## 5. 非功能需求

### NFR-01：性能要求
- 响应时间 < X秒
- 支持X个并发

### NFR-02：安全要求
- XSS防护
- 内容过滤

## 6. 验收标准

### AC-01：[验收项]
- **Given**：[前置条件]
- **When**：[执行操作]
- **Then**：[预期结果]
- **优先级**：P0/P1/P2

### AC-02：[验收项]
[同上]

## 7. 边界条件
- **最大值**：[限制]
- **最小值**：[限制]
- **特殊情况**：[处理方式]

## 8. 异常处理
| 异常情况 | 处理方式 | 用户提示 |
|---------|---------|---------|
| [异常1] | [处理] | [提示] |
| [异常2] | [处理] | [提示] |

## 9. 数据需求
- **输入数据**：[需要哪些输入]
- **输出数据**：[返回哪些数据]
- **存储数据**：[需要存储什么]

## 10. 约束条件
- **技术约束**：[技术限制]
- **时间约束**：[开发时间]
- **资源约束**：[人力/预算]
```

### 4. 确认需求文档
向用户展示关键需求点：
```
已生成《[功能名称]功能需求》，关键点：

用户角色：注册用户可评论
核心场景：文章详情页发表评论和回复
验收标准：X个（已列出）

有一些细节需要确认：
[列出需要确认的点]

确认后我保存到 specs/features/[功能名称].md
```

### 5. 保存文档
将功能需求文档保存到 `specs/features/[功能名称].md`

## 输出格式
- 文件路径：`specs/features/[功能名称].md`
- 文档格式：标准化的功能需求文档
- 内容要求：具体、可测试、有优先级

## 注意事项

### 边界守卫规则
- **当前阶段只能做**：功能需求澄清和验收标准定义
- **禁止行为**：不涉及技术方案设计、不涉及代码实现
- 如果用户问"怎么实现"或"用什么技术"，提醒："我们先专注于想清楚要什么，技术方案的事后面会处理"

### 项目上下文协议
- 执行前读取项目文档了解上下文
- 功能需求要与产品概述保持一致

### 提问技巧
- 全程不出现技术术语
- 每个问题都提供选项和推荐
- 问题要具体，避免开放式大问题
- 根据用户回答调整后续提问

### 验收标准编写
- 使用 Given-When-Then 格式
- 每个标准可测试、可验证
- 标注优先级（P0/P1/P2）

## 使用示例

**用户输入**：
```
/feature-requirements-clarification 我想做一个评论功能
```

**智能体响应**：
```
好的，我来帮你把评论功能的需求理清楚。先从场景开始聊：

评论主要出现在什么地方？

A：文章详情页（读者对文章发表看法）
B：商品页面（买家评价商品）
C：其他场景

推荐：A，因为根据产品概述，你做的是内容平台

你先告诉我这个，我们再往下聊。
```

**后续对话示例**：
```
用户：选A

智能体：好的，文章详情页的评论。下一个问题：

谁能评论？

A：所有访客（无需登录）
B：注册用户（需要登录）
C：特定权限用户

推荐：B，既能防止垃圾评论，又能建立用户体系

评论能做什么？

A：只能发表评论
B：可以回复评论（嵌套）
C：可以点赞评论
D：以上全部

推荐：D，完整功能，但我们可以分阶段实现
...
```
