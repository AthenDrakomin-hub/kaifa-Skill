# 任务规划

## 任务目标
- 本Skill用于：将技术方案拆解为可执行的开发任务
- 能力包含：任务拆分、依赖分析、时间估算、风险评估
- 触发条件：已完成技术方案设计，准备开始编码实现

## 前置准备
- 必须存在 `specs/features/[功能名称]_技术方案.md`
- 建议存在 `specs/features/[功能名称].md`
- 工作目录：项目根目录

## 操作步骤

### 1. 读取前置文档
智能体扮演技术主管角色，读取必要的文档：
```
读取 specs/features/[功能名称]_技术方案.md
读取 specs/features/[功能名称].md（如果存在）
读取 specs/开发规范.md（如果存在）
```

### 2. 分析技术方案
识别技术方案中的关键模块：
- **数据层**：数据库表、数据访问层
- **接口层**：API接口、业务逻辑
- **表现层**：UI组件、用户交互
- **测试层**：单元测试、集成测试

### 3. 拆分开发任务
按模块和依赖关系拆分任务：

```markdown
# [功能名称]任务规划

## 1. 任务拆分

### 第1阶段：数据层
预计工时：105分钟

#### Task-01：创建 comments 表及索引
- **预计工时**：30分钟
- **涉及文件**：
  - `migrations/003_create_comments.sql`
- **依赖**：无
- **验收标准**：
  - [ ] 表结构正确创建
  - [ ] 索引正确创建
  - [ ] 外键约束正常工作
- **风险**：低

#### Task-02：实现评论数据访问层 CRUD
- **预计工时**：45分钟
- **涉及文件**：
  - `src/modules/comments/services/commentRepository.ts`
- **依赖**：Task-01
- **验收标准**：
  - [ ] create 方法正常工作
  - [ ] findById 方法正常工作
  - [ ] findByPostId 方法正常工作
  - [ ] update 方法正常工作
  - [ ] delete 方法正常工作

#### Task-03：数据层单元测试
- **预计工时**：30分钟
- **涉及文件**：
  - `src/modules/comments/services/__tests__/commentRepository.test.ts`
- **依赖**：Task-02
- **验收标准**：
  - [ ] 所有测试用例通过
  - [ ] 覆盖率 > 80%
- **风险**：低

### 第2阶段：接口层

  - [ ] 参数验证正确
  - [ ] 错误处理完善
  - [ ] 返回格式正确

#### Task-05：实现获取评论列表 API（含嵌套）
- **预计工时**：50分钟
- **涉及文件**：
  - `src/app/api/comments/route.ts`（更新）

#### Task-06：接口层集成测试

- **涉及文件**：
  - `src/app/api/comments/__tests__/route.test.ts`
- **依赖**：Task-05
- **验收标准**：
  - [ ] 所有测试用例通过
  - [ ] 覆盖率 > 80%

### 第3阶段：表现层

#### Task-07：评论列表组件
- **预计工时**：50分钟
- **涉及文件**：

- **依赖**：Task-05
- **验收标准**：
  - [ ] 评论列表正确渲染
  - [ ] 嵌套回复正确展示
  - [ ] 分页功能正常
  - [ ] 样式符合设计

#### Task-08：发布评论组件
- **预计工时**：40分钟
- **涉及文件**：
  - `src/modules/comments/components/CommentForm.tsx`
  - `src/modules/comments/components/ReplyForm.tsx`
- **依赖**：Task-04
- **验收标准**：
  - [ ] 表单验证正确
  - [ ] 提交功能正常
  - [ ] 错误提示友好
  - [ ] 样式符合设计
- **风险**：低

## 2. 任务依赖图

```
Task-01 (创建表)
    ↓
Task-02 (数据访问层) → Task-03 (数据层测试)
    ↓                      ↓
Task-04 (发布API) ─────────┼→ Task-05 (获取API) → Task-06 (接口测试)
    ↓                      ↓
Task-08 (发布组件)         Task-07 (列表组件)
```

## 3. 关键路径
- **关键路径**：Task-01 → Task-02 → Task-04 → Task-05 → Task-07
- **关键路径工时**：175分钟
- **总工时**：320分钟

## 4. 风险评估

### 高风险任务
- 无

### 中风险任务
- Task-04：发布评论 API
  - 风险：权限验证逻辑复杂
  - 应对：提前确认权限方案

- Task-05：获取评论列表 API
  - 风险：嵌套查询性能问题
  - 应对：限制嵌套层数，优化查询

### 阻塞任务
- Task-01：阻塞所有后续任务
- Task-02：阻塞接口层和表现层

## 5. 执行建议
- 按阶段顺序执行，每完成一个阶段进行验证
- 每个任务完成后立即编写测试
- 中风险任务优先安排，留出调试时间

## 6. 进度追踪
- [ ] 第1阶段：数据层（3个任务）
- [ ] 第2阶段：接口层（3个任务）
- [ ] 第3阶段：表现层（2个任务）

**当前进度**：0% (0/8 任务完成)
```

### 4. 保存任务规划文档
将任务规划保存到 `specs/features/[功能名称]_任务规划.md`

### 5. 确认任务规划
向用户展示任务规划：
```
已读取技术方案和需求文档，拆解出 8 个开发任务，分 3 个阶段，预计总工时 320 分钟。

第 1 阶段：数据层（3 个任务）

Task-01：创建 comments 表及索引 [30min]

Task-05：实现获取评论列表 API（含嵌套）[50min] ← 依赖 Task-02

Task-07：评论列表组件 [50min] ← 依赖 Task-05

需要调整优先级或合并/拆分某些任务吗？确认后我生成任务规划文档。
```

## 输出格式
- 文件路径：`specs/features/[功能名称]_任务规划.md`
- 文档格式：结构化的任务规划文档
- 内容要求：具体任务、明确依赖、可追踪进度

## 注意事项

### 边界守卫规则
- **当前阶段只能做**：任务拆分和规划
- **禁止行为**：不涉及具体代码编写
- 如果用户说"开始执行任务"，提醒："任务规划好后，我们会在编码实现阶段逐个执行"

### 项目上下文协议
- 执行前必须读取 `specs/features/[功能名称]_技术方案.md`
- 任务拆分要符合技术方案的设计

### 任务拆分原则
- 每个任务控制在2小时以内
- 单一职责：一个任务只做一件事
- 明确验收标准
- 标注风险等级
- 理清依赖关系
