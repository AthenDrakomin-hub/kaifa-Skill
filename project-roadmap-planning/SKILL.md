# 路线图规划

## 任务目标
- 本Skill用于：分析模块依赖关系并制定开发顺序
- 能力包含：依赖分析、里程碑划分、任务拆解、风险识别
- 触发条件：已完成开发规范制定，需要规划开发路线

## 前置准备
- 必须存在 `specs/产品概述.md`
- 必须存在 `specs/技术栈.md`
- 必须存在 `specs/项目结构.md`
- 工作目录：项目根目录

## 操作步骤

### 1. 读取前置文档
智能体扮演技术产品经理角色，读取必要的文档：
```
读取 specs/产品概述.md
读取 specs/技术栈.md
读取 specs/项目结构.md
```

### 2. 分析模块依赖
分析各模块之间的依赖关系：

**依赖类型**：
- **硬依赖**：A模块必须在B模块之前完成
- **软依赖**：A模块可以在B模块之后或并行开发
- **无依赖**：模块之间没有直接关系

**依赖分析维度**：
- 数据依赖：模块A使用模块B的数据
- 功能依赖：模块A调用模块B的功能
- 基础设施依赖：所有业务模块依赖基础设施

### 3. 划分里程碑
基于依赖关系划分开发阶段：

```markdown
# 开发路线图

## 1. 模块依赖分析

### 依赖关系图
```
基础设施（auth, shared）
    ↓
核心业务（posts）
    ↓
前台展示（blog-pages）
    ↓
增强功能（analytics, admin）
```

### 模块依赖表
| 模块 | 依赖模块 | 依赖类型 | 说明 |
|------|----------|----------|------|
| posts | auth | 硬依赖 | 文章需要用户认证 |
| analytics | posts | 硬依赖 | 统计需要文章数据 |
| admin | auth | 硬依赖 | 管理后台需要认证 |

## 2. 里程碑规划

### 里程碑一：MVP（核心闭环）
**目标**：实现最小可用产品
**预计工期**：X周

#### Phase 1.1：基础设施
- [ ] 用户认证模块（auth）
  - 登录/注册功能
  - Session管理
  - 权限控制
- [ ] 公共组件库（shared）
  - 基础UI组件
  - 工具函数
  - 类型定义

#### Phase 1.2：核心业务
- [ ] 文章管理（posts）
  - 文章创建/编辑
  - Markdown渲染
  - 文章发布/下线

#### Phase 1.3：前台展示
- [ ] 博客前台页面
  - 文章列表页
  - 文章详情页
  - 标签筛选

### 里程碑二：完整版
**目标**：完善核心功能
**预计工期**：X周

#### Phase 2.1：阅读统计
- [ ] 统计分析模块（analytics）
  - 阅读量统计
  - 访问趋势
  - 数据可视化

#### Phase 2.2：后台管理
- [ ] 管理后台（admin）
  - 文章管理界面
  - 用户管理
  - 数据统计看板

### 里程碑三：增强版
**目标**：优化和扩展
**预计工期**：X周

#### Phase 3.1：SEO优化
- [ ] SEO功能
  - Sitemap生成
  - Open Graph标签
  - 结构化数据

#### Phase 3.2：性能优化
- [ ] 性能优化
  - 图片懒加载
  - 代码分割
  - 缓存策略

## 3. 开发顺序建议

### 推荐开发路径
```
Phase 1.1 → Phase 1.2 → Phase 1.3 → Phase 2.1 → Phase 2.2 → Phase 3.1 → Phase 3.2
```

### 关键路径
- **关键路径**：auth → posts → blog-pages
- **阻塞任务**：auth模块未完成会阻塞所有业务模块

## 4. 风险评估

### 高风险项
- [ ] 第三方服务集成（如数据库）
  - 风险：服务不稳定或配置复杂
  - 应对：提前测试、准备备选方案

### 中风险项
- [ ] 性能瓶颈
  - 风险：数据量增长后性能下降
  - 应对：早期设计考虑扩展性

- [ ] Phase 1.2 - 完成日期：YYYY-MM-DD
- [ ] Phase 1.3 - 完成日期：YYYY-MM-DD
...
```

## 输出格式

## 注意事项

### 边界守卫规则
- **当前阶段只能做**：开发路线规划和里程碑划分
- **禁止行为**：不涉及具体代码编写、不涉及任务执行
- 如果用户问"开始做Phase 1.1"，提醒："路线图规划好后，我们会在项目初始化阶段创建项目骨架"

### 项目上下文协议
- 执行前必须读取 `specs/产品概述.md`、`specs/技术栈.md`、`specs/项目结构.md`
- 基于产品需求和技术架构规划开发顺序

### 规划原则
- 先做地基，再做核心，最后做增强
- 识别阻塞任务并优先安排
- 每个Phase工作量适中（建议1-2周）
