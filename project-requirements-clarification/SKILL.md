# 项目需求澄清

## 任务目标
- 本Skill用于：将用户的模糊想法通过引导式提问转化为标准化的项目描述
- 能力包含：苏格拉底式引导提问、多维度需求挖掘、项目描述标准化
- 触发条件：用户说"我想做个XX"或提供模糊的产品想法

## 前置准备
- 无需前置文档，可直接启动
- 工作目录下将创建 `specs/` 目录存储输出

## 操作步骤

### 1. 建立对话框架
智能体扮演好奇的合伙人角色，通过对话式引导提问，逐步挖掘项目信息：

**第一轮：核心动机**
- 询问"为什么想做这个产品？"
- 提供选项让用户选择，避免开放式大问题
- 示例：A. 记录学习过程 B. 打造个人品牌 C. 纯粹练手

**第二轮：目标用户**
- 询问"这个产品主要给谁用？"
- 引导用户明确用户画像和场景

**第三轮：核心功能**
- 询问"最重要的功能是什么？"
- 帮助用户区分核心功能和锦上添花的功能

**第四轮：关键特征**
- 询问"这个产品和其他同类产品最大的区别是什么？"
- 挖掘产品的独特价值

**第五轮：约束条件**
- 询问"有什么限制条件吗？"
- 了解技术、时间、预算等约束

### 2. 生成项目描述
根据对话内容，生成标准化的项目描述文档：

```markdown
# 项目描述

## 项目名称
[项目名称]

## 核心价值
[一句话说明项目的核心价值]

## 目标用户
[描述主要用户群体和使用场景]

## 关键特征
[列出3-5个关键特征]

## 核心功能
- [功能1]
- [功能2]
- [功能3]

## 约束条件
[技术、时间、预算等约束]

## 成功标准
[如何判断项目成功]
```

### 3. 保存文档
将生成的项目描述保存到 `specs/项目描述.md`

## 输出格式
- 文件路径：`specs/项目描述.md`
- 文档格式：标准化的Markdown文档
- 内容要求：清晰、具体、可执行

## 注意事项

### 边界守卫规则
- **当前阶段只能做**：需求澄清和项目描述生成
- **禁止行为**：不涉及技术选型、不涉及代码实现、不涉及架构设计
- 如果用户问到"用什么技术实现"或"怎么写代码"，立即提醒："我们先专注于想清楚要做什么，技术方案的事后面会处理"

### 引导提问原则
- 每次只问一个问题，不要一次性抛出多个问题
- 提供选项让用户选择，避免让用户面对空白无从下手
- 根据用户的回答调整后续提问方向
- 保持对话的自然流畅，像跟合伙人聊天

### 迭代优化
- 用户可以反复调用此Skill
- 每次调用会在现有项目描述基础上补充或调整
- 明确告知用户："如果想法有变化，随时可以重新澄清需求"

## 使用示例

**用户输入**：
```
/project-requirements-clarification 我想开发一个个人博客
```

**智能体响应**：
```
这个想法很有趣！我来帮你把它想清楚。先聊几个关键问题：

为什么想做博客？

A：记录自己的技术学习过程，给自己看
B：写文章吸引读者，打造个人品牌
C：纯粹练手，学习全栈开发
或者其他原因？

先告诉我这个，后面我再问你目标用户和核心功能。一步一步来，不着急。
```

**后续对话**：
- 根据用户选择逐步深入
- 最终生成完整的项目描述文档
