# 项目结构设计

## 任务目标
- 本Skill用于：设计项目的目录结构和文件组织方式
- 能力包含：模块划分、目录设计、文件布局、架构模式选择
- 触发条件：已完成技术栈选型，需要规划代码结构

## 前置准备
- 必须存在 `specs/产品概述.md`
- 必须存在 `specs/技术栈.md`
- 工作目录：项目根目录

## 操作步骤

### 1. 读取前置文档
智能体扮演系统架构师角色，读取必要的文档：
```
读取 specs/产品概述.md
读取 specs/技术栈.md
```

### 2. 分析系统模块
基于产品概述中的核心板块，映射为代码模块：

**模块映射原则**：
- 每个核心板块对应一个独立模块
- 公共功能提取到共享模块
- 第三方服务封装为独立层

### 3. 设计目录结构
根据技术栈的最佳实践设计目录：

```markdown
# 项目结构设计

## 1. 设计原则
- **高内聚**：相关代码放在同一模块
- **低耦合**：模块之间通过清晰接口交互
- **可扩展**：便于后续添加新功能
- **可维护**：代码组织清晰，易于定位

## 2. 目录结构

```
project-name/
├── src/
│   ├── app/                  # 应用入口和路由
│   │   ├── (auth)/          # 认证相关页面
│   │   ├── (blog)/          # 博客前台页面
│   │   └── admin/           # 后台管理页面
│   ├── modules/             # 业务模块（核心）
│   │   ├── auth/            # 用户认证模块
│   │   │   ├── components/  # 模块组件
│   │   │   ├── services/    # 业务逻辑
│   │   │   └── types/       # 类型定义
│   │   ├── posts/           # 文章管理模块
│   │   └── analytics/       # 统计分析模块
│   ├── shared/              # 公共组件和工具
│   │   ├── components/      # 通用组件
│   │   ├── hooks/           # 通用Hooks
│   │   └── utils/           # 工具函数
│   └── lib/                 # 第三方服务封装
│       ├── database/        # 数据库客户端
│       └── api/             # API客户端
├── specs/                   # 项目规格文档
├── docs/                    # 开发文档
│   └── dev-logs/           # 开发记录
├── public/                  # 静态资源
├── .env.example            # 环境变量示例
├── .gitignore              # Git忽略规则
└── README.md               # 项目说明
```

## 3. 模块职责说明

### app/ 目录
[说明路由组织方式和页面结构]

### modules/ 目录
[说明每个模块的职责和内部结构]

### shared/ 目录
[说明公共组件和工具的组织方式]

### lib/ 目录
[说明第三方服务的封装方式]

## 4. 命名约定
- **目录名**：使用 kebab-case（如 user-profile/）
- **组件文件**：使用 PascalCase（如 PostCard.tsx）
- **工具函数**：使用 camelCase（如 formatDate.ts）
- **类型文件**：使用 PascalCase（如 User.ts）

## 5. 文件组织示例

### 模块结构示例
```
modules/posts/
├── components/
│   ├── PostCard.tsx        # 文章卡片组件
│   ├── PostEditor.tsx      # 文章编辑器
│   └── index.ts            # 组件导出
├── services/
│   ├── postService.ts      # 文章服务
│   └── index.ts            # 服务导出
├── types/
│   ├── Post.ts             # 文章类型
│   └── index.ts            # 类型导出
└── index.ts                # 模块导出
```
```

### 4. 确认结构方案
向用户展示设计方案并确认：
```
设计思路：采用特性架构（Feature-based），每个 modules/ 下的模块是独立的业务单元。后续开发时，AI 只需要聚焦到对应的模块目录即可。

你觉得这个结构 OK 吗？有需要调整的地方告诉我。
```

### 5. 保存文档
将项目结构设计保存到 `specs/项目结构.md`

## 输出格式
- 文件路径：`specs/项目结构.md`
- 文档格式：结构化的目录设计文档
- 内容要求：清晰的目录树、明确的模块职责、具体的命名约定

## 注意事项

### 边界守卫规则
- **当前阶段只能做**：目录结构设计和文件组织规划
- **禁止行为**：不涉及具体代码编写、不涉及数据库表设计
- 如果用户问"某个模块的代码怎么写"，提醒："我们先把目录结构定好，具体的代码实现后面会处理"

### 项目上下文协议
- 执行前必须读取 `specs/产品概述.md` 和 `specs/技术栈.md`
- 基于产品模块和技术栈最佳实践进行设计

### 设计原则
- 遵循所选技术栈的最佳实践
- 考虑后续开发的便利性
- 便于AI理解和操作（目录结构清晰、职责明确）
