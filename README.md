# AI辅助开发工作流系统

一套完整的AI辅助开发工作流系统，包含13个专业Skill，帮助没有编程基础的用户也能高效开发项目。

## 📋 系统概述

本系统将软件开发流程系统化为13个独立Skill，分为三个层次：

- **项目级**：启动时执行一次，搭建项目骨架
- **功能级**：反复使用，逐个功能开发
- **通用级**：随时调用，处理异常情况

## 🎯 核心机制

### 1. 边界守卫（GUARDRAILS）
每个Skill有明确的职责边界，强制AI在正确的时间做正确的事：
- 需求阶段只能聊需求
- 设计阶段只能出方案
- 编码阶段才能写代码

### 2. 项目上下文协议（PROJECT-CONTEXT）
每个Skill执行前强制读取`specs/`目录文档：
- AI不会忘记项目背景
- 开新窗口也能继续工作
- 保证技术方案的一致性

## 📦 Skill清单

### 项目级工作流（执行一次）

#### 1. project-requirements-clarification
**需求澄清** - 将模糊想法转化为标准化项目描述

- **核心能力**：苏格拉底式引导提问、多维度需求挖掘
- **触发场景**：用户说"我想做个XX"
- **输出文档**：`specs/项目描述.md`

**示例**：
```
用户：我想开发一个个人博客
AI：这个想法很有趣！先聊几个关键问题：
为什么想做博客？
A：记录学习过程
B：打造个人品牌
C：纯粹练手
```

#### 2. project-product-overview
**产品概述** - 将聊天式需求升级为正式产品文档

- **核心能力**：产品愿景定义、验收标准制定
- **触发场景**：需求澄清完成后
- **输出文档**：`specs/产品概述.md`

#### 3. project-tech-stack
**技术栈选型** - 推荐最合适的技术组合

- **核心能力**：约束分析、方案对比、技术决策支持
- **触发场景**：产品概述确定后
- **输出文档**：`specs/技术栈.md`

**特点**：推荐"最合适"而非"最热门"，考虑用户技术背景

#### 4. project-structure
**项目结构** - 设计高内聚低耦合的目录结构

- **核心能力**：模块划分、文件布局、架构模式选择
- **触发场景**：技术栈确定后
- **输出文档**：`specs/项目结构.md`

#### 5. project-dev-standards
**开发规范** - 制定代码风格、命名约定、Git工作流

- **核心能力**：规范定义、AI协作协议制定
- **触发场景**：项目结构设计后
- **输出文档**：`specs/开发规范.md`

#### 6. project-roadmap-planning
**路线图规划** - 分析依赖关系，划分开发里程碑

- **核心能力**：依赖分析、里程碑划分、风险识别
- **触发场景**：规范制定后
- **输出文档**：`specs/开发路线图.md`

#### 7. project-initialization
**项目初始化** - 严格按照文档创建项目骨架

- **核心能力**：目录创建、依赖安装、配置生成
- **触发场景**：所有规划完成后
- **输出文档**：项目骨架 + `docs/开发记录/初始化记录.md`

### 功能级工作流（反复执行）

#### 8. feature-requirements-clarification
**功能需求澄清** - 深入挖掘单个功能的需求细节

- **核心能力**：功能边界定义、验收标准制定
- **触发场景**：开始开发某个具体功能
- **输出文档**：`specs/features/[功能名称].md`

**特点**：全程不出现技术术语，只聊用户要什么

#### 9. feature-tech-design
**技术方案设计** - 设计API接口、数据库表、核心逻辑

- **核心能力**：接口设计、数据库设计、方案对比
- **触发场景**：功能需求明确后
- **输出文档**：`specs/features/[功能名称]_技术方案.md`

#### 10. feature-task-planning
**任务规划** - 将技术方案拆解为可执行的小任务

- **核心能力**：任务拆分、依赖分析、时间估算
- **触发场景**：技术方案确定后
- **输出文档**：`specs/features/[功能名称]_任务规划.md`

**特点**：每个任务控制在2小时内，单一职责，明确验收标准

#### 11. feature-implementation
**编码实现** - 按任务规划逐个执行

- **核心能力**：代码编写、测试执行、进度追踪
- **触发场景**：任务规划完成后
- **输出文档**：代码文件 + 阶段完成报告

**特点**：一个阶段一个阶段执行，每个任务完成后立即写测试

#### 12. feature-evolution
**功能迭代变更** - 处理已完成功能的修改需求

- **核心能力**：变更影响分析、增量文档更新
- **触发场景**：需要对已完成功能进行修改
- **输出文档**：变更记录 + 新任务

**变更类型**：
- **Tweak（微调）**：小改动，不涉及新需求
- **Extension（扩展）**：加新东西，需要新增验收标准
- **Refactor（重构）**：动了核心结构

### 通用级工作流（随时使用）

#### 13. bugfix-workflow
**BUG修复** - 强制执行完整修复流程

- **核心能力**：问题复现、根因定位、代码修复、回归测试
- **触发场景**：遇到BUG问题
- **输出文档**：`docs/BUG修复文档/BUG-XXX.md`

**硬性规则**：🚨 **未复现，不修复**

## 🔄 完整使用流程

### 新项目启动流程

```
第1步：需求澄清
→ /project-requirements-clarification "我想做个XX"
→ 生成 specs/项目描述.md

第2步：产品概述
→ /project-product-overview
→ 生成 specs/产品概述.md

第3步：技术选型
→ /project-tech-stack
→ 生成 specs/技术栈.md

第4步：结构设计
→ /project-structure
→ 生成 specs/项目结构.md

第5步：规范制定
→ /project-dev-standards
→ 生成 specs/开发规范.md

第6步：路线规划
→ /project-roadmap-planning
→ 生成 specs/开发路线图.md

第7步：项目初始化
→ /project-initialization
→ 创建项目骨架和配置文件
```

### 功能开发流程（循环执行）

```
第1步：功能需求
→ /feature-requirements-clarification "我想做个XX功能"
→ 生成 specs/features/XX功能.md

第2步：技术方案
→ /feature-tech-design XX功能
→ 生成 specs/features/XX功能_技术方案.md

第3步：任务规划
→ /feature-task-planning XX功能
→ 生成 specs/features/XX功能_任务规划.md

第4步：编码实现
→ /feature-implementation 完成XX功能的任务 第N阶段
→ 执行任务、编写测试、生成报告

第5步：（如需修改）
→ /feature-evolution "我想给XX功能加个YY"
→ 更新文档、生成新任务
```

### BUG修复流程

```
→ /bugfix-workflow "描述BUG现象"
→ AI追问细节 → 复现问题 → 定位根因 → 修复代码 → 验证测试
→ 生成 docs/BUG修复文档/BUG-XXX.md
```

## 💡 设计理念

### 为什么不需要脚本？

这套工作流的所有任务都属于：
- ✅ 内容创作：需求文档、技术方案
- ✅ 分析推理：依赖分析、影响评估
- ✅ 知识咨询：技术选型建议
- ✅ 代码生成：AI已具备的能力

根据**实现方式选型指南**，这些能力智能体已经具备，无需脚本。

### 为什么这么设计？

**1. 系统性规划**
- 从模糊想法到清晰方案，层层递进
- 避免做到一半发现需求没想清楚

**2. 约束AI行为**
- 边界守卫防止AI越界
- 项目上下文协议防止AI遗忘

**3. 降低使用门槛**
- 引导式提问而非填表
- 每步都有选项和推荐
- 技术术语只在必要时出现

**4. 保证代码质量**
- 每个功能都有验收标准
- 每个任务都写测试
- 每次修改都留记录
